## 👤 主人偏好模式

- **简洁优先**: 主人喜欢直接、简洁的回答
- **选项驱动**: 喜欢看到选项然后选择，而不是被一步步引导

## ✅ 成功模式库

### 早报内容模式
- **来源**: 2026-03-14 早报多次教训
- **规则**:
  1. **早报 = 回顾昨天**，不是预告今天
  2. **天气**：今日天气（告诉主人今天穿什么、带不带伞）
  3. **昨日工作**：昨天的日志内容
  4. **新闻**：昨日的60秒新闻（回顾昨天发生的大事）
     - **必须完整列出所有条目**，不能遗漏
     - 标题：📰 昨日新闻回顾
  5. **微语**：直接写"💡 微语："，不需要强调"昨日"
  6. **日期计算**：昨天 = 今天 - 1（按主人时区UTC+8）
- **新闻处理**：
  ```
  API返回15条新闻 → 全部列出，不能遗漏
  微语 → 💡 微语：内容（不加"昨日"）
  ```

### 日期计算模式
- **来源**: 2026-03-14 早报日期错误教训
- **规则**: 
  1. **先获取当前时间**：用 `session_status` 获取当前日期
  2. **动态计算目标日期**：昨天 = 今天 - 1，不要用缓存或硬编码
  3. **注意时区**：主人时区 UTC+8，系统时区 UTC，日期判断要用主人时区
- **应用场景**：早报、日报、周报、日志读取等所有涉及日期的场景
- **示例**：
  ```
  当前 UTC: 2026-03-14 01:00
  主人时区(UTC+8): 2026-03-14 09:00
  昨天(按主人时区): 2026-03-13
  应读取: memory/2026-03-13.md
  ```

### 长任务处理模式
- **来源**: 2026-03-13 企业微信消息超时问题
- **问题**: 复杂任务执行时间超过 6 分钟，企业微信流式消息过期无法发送
- **模式**: 
  1. **先确认**：收到耗时任务时，先回复"正在处理中，请稍候..."确认收到
  2. **分批执行**：将大任务拆分成小任务，每完成一步汇报进度
  3. **控制时间**：单个任务执行控制在 3 分钟内，超时先汇报进度再继续
- **应用**: 安装依赖、创建多个文件、批量操作等耗时超过 30 秒的任务

### 多能力建设模式
- **来源**: 2026-03-13 五大能力补齐
- **模式**: 主人给出能力清单时，一次性完成所有能力的建设，包括：skill 文件 + 数据文件 + 配置文件 + 依赖安装
- **应用**: 收到多个能力需求时，创建完整的能力栈，而非仅创建文档

### 系统设计模式
- **来源**: 2026-03-13 自我进化系统设计
- **模式**: 主人偏好系统性方案，当有多个改进选项时，倾向于"全部实施"
- **应用**: 涉及系统改进时，主动提供完整方案
