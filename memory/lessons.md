### 早报格式混乱 - 多处日期错误且缺少昨日新闻
- **时间**: 2026-03-15
- **问题**: 
  1. 开头日期写错 - 应写当天日期（3月15日），却写成了其他日期
  2. 昨日任务日期计算错误 - 应该是今天-1=3月14日，却写成了3月13日
  3. 完全没有加入昨日新闻 - 早报必须包含昨日的60秒新闻和微语
- **原因**: 
  1. 早报格式没有明确规范，各部分日期来源不清晰
  2. 动态日期计算错误，没有正确使用 session_status 获取当前日期
  3. 对"早报"的理解不完整，缺少关键组成部分
- **解决**: 
  1. 制定详细的《早报格式规范》文档，明确每部分的日期来源
  2. 早报 = 开头(当天日期) + 昨日任务(今天-1) + 昨日新闻(今天-1) + 昨日微语(今天-1)
  3. 每次生成早报前，先用 session_status 获取当天日期，再计算昨天日期
- **预防**: 严格遵守《早报格式规范》，生成前检查日期计算
- **标签**: #早报 #格式规范 #日期计算 #严重格式错误

### 早报缺少天气和注意事项
- **时间**: 2026-03-15
- **问题**: 早报开头漏掉了天气和注意事项，主人再次指出
- **原因**: 
  1. patterns.md 已记录早报应有天气，但未写入《早报格式规范》
  2. 生成早报时没有检查 patterns.md
  3. 教训记录后没有及时更新规范文档
- **解决**: 
  1. 更新《早报格式规范》，加入天气和注意事项
  2. 早报完整格式：天气 + 注意事项 + 昨日任务 + 昨日新闻 + 微语
- **预防**: 每次生成早报前，检查 patterns.md 和《早报格式规范》
- **标签**: #早报 #天气 #格式规范 #遗漏

### 早报新闻日期错误 - 发了今日新闻而非昨日新闻
- **时间**: 2026-03-14
- **问题**: 早报中发了"今日60秒新闻"，但主人想看的是"昨日60秒新闻"回顾昨天发生的大事
- **原因**: 
  1. 没有理解早报的使用场景：早上发报，主人想回顾昨天发生的事情
  2. 直接获取今日新闻，而非昨天的新闻
- **解决**: 
  1. 早报中的新闻应该是昨天的日期
  2. 获取新闻时：`curl ... &date=昨天日期`
  3. 标题改为"昨日60秒新闻回顾"
- **预防**: 早报内容应该是"回顾昨天"，不是"预告今天"
- **标签**: #早报 #新闻 #日期 #逻辑错误

### 早报内容不完整和微语表述错误
- **时间**: 2026-03-14
- **问题**: 
  1. 昨日新闻只列了11条，实际有15条，内容不完整
  2. 微语写"昨日微语"，但应该直接写"微语"即可
- **原因**: 
  1. 新闻条目没有全部列出，遗漏了4条
  2. 过度强调"昨日"，微语本身不需要时间限定
- **解决**:
  1. 新闻必须完整列出，不能遗漏
  2. 微语直接写"💡 微语："或"💡 今日微语："
- **标签**: #早报 #内容完整 #表述

### 早报日期计算错误 - 用了错误日期的日志
- **时间**: 2026-03-14
- **问题**: 早报中"昨日工作"总结了 3月12日 而不是 3月13日 的内容，日期计算错误
- **原因**: 
  1. 没有动态计算"昨天"日期，可能用了缓存或硬编码
  2. 没有考虑时区差异（主人 UTC+8，系统 UTC）
- **解决**: 
  1. 早报生成时，必须用 `session_status` 获取当前日期，然后计算昨天的日期
  2. 昨天日期 = 当前日期 - 1（按主人时区 UTC+8）
  3. 读取正确的 `memory/YYYY-MM-DD.md` 文件
- **预防**: 每次涉及日期计算时，先获取当前时间，再动态计算目标日期
- **标签**: #日期计算 #早报 #时区 #错误

### OpenClaw Hook 不支持动态 workspace 创建

### OpenClaw 配置 denyCommands 无效
- **时间**: 2026-03-12
- **问题**: 主人在远端配置了 `tools.exec.denyCommands`，但该配置项不被 OpenClaw 识别
- **解决**: 运行 `openclaw doctor --fix` 移除无效配置
- **标签**: #openclaw #config

### 飞书消息长度限制 - 4096 字符上限
- **时间**: 2026-03-17
- **问题**: 飞书插件不会自动分割超长消息，超过 4096 字符会发送失败
- **原因**: 飞书 API 对文本/富文本消息有 4096 字符限制，需要手动拆分
- **解决**: 
  1. 长消息主动拆分成多段，每段控制在 4000 字符以内
  2. 或使用文档链接代替长消息
  3. 或改用支持自动分段的渠道（如企业微信）
- **预防**: 发送前检查消息长度，超过 3500 字符时主动拆分
- **标签**: #飞书 #消息长度 #限制
